# 🔒 CloudSync Edge — агент фонового шифрования и синхронизации файлов

Лёгковесный Python-агент, который следит за рабочей папкой в реальном времени и при любом изменении файла мгновенно создаёт его зашифрованную копию в целевом хранилище. Работает по событийной модели (через `watchdog`), без постоянного опроса диска — никаких циклов `while True: scan_folder()`.

Проект — это **рабочее ядро (core)** системы синхронизации с шифрованием. Он демонстрирует архитектуру и базовую логику; для продакшн-использования в компании ядро дорабатывается под конкретную инфраструктуру (см. раздел «Коммерческая доработка» ниже).

---

## 🧠 Как это работает

1. **Мониторинг файловой системы.** `watchdog` подключается к ядру ОС и получает уведомления об изменениях в указанной директории — без активного сканирования папок.
2. **Шифрование перед отправкой.** При изменении файла его содержимое читается, шифруется симметричным алгоритмом Fernet (AES-128-CBC + HMAC) из библиотеки `cryptography`, и сохраняется как `<имя_файла>.enc`.
3. **Запись в целевую директорию.** Зашифрованная копия попадает в папку, которая в демо-версии играет роль "облака" — на деле это обычная локальная директория. Исходный файл на месте не изменяется.

---

## 🚀 Что есть в текущей версии

- **Событийная архитектура** — агент активируется только при реальном изменении файла, низкая нагрузка на систему в простое.
- **Шифрование на лету** — каждое изменение файла превращается в зашифрованный `.enc`-объект средствами `cryptography.fernet`.
- **Работа в отдельном потоке** — мониторинг запускается как daemon-поток и не блокирует основной процесс.
- **Автоматическое создание директорий** — при первом запуске создаются папки `MyDocuments` (исходники) и `NetworkCloudStorage` (демо-хранилище), если их нет.

## ⚠️ Важные ограничения текущей версии (демо/ядро)

Чтобы не создавать ложных ожиданий, честно: то, что есть в репозитории, — это **прототип логики**, а не готовое решение корпоративного уровня. В частности:

- **Ключ шифрования не сохраняется.** Он генерируется заново при каждом запуске (`Fernet.generate_key()`) и хранится только в памяти процесса. После остановки скрипта расшифровать ранее созданные `.enc`-файлы **не получится** — ключ нигде не сохранён. Это нужно дорабатывать отдельно (безопасное хранение/ротация ключей) для любого реального использования.
- **Нет модуля расшифровки.** В демо есть только прямое направление (файл → шифротекст). Восстановление данных из `.enc` нужно реализовывать отдельно.
- **Отслеживается только изменение существующих файлов** (`on_modified`). Создание нового файла или его перемещение могут не вызвать немедленную синхронизацию — это можно расширить, добавив обработку `on_created` и `on_moved`.
- **Без вложенных папок** — мониторинг не рекурсивный (`recursive=False`), подпапки не отслеживаются.
- **"Облако" — это локальная папка.** Реальной сетевой передачи (SFTP/FTP/S3 и т.п.) в этой версии нет — выгрузка эмулируется записью на тот же диск.
- **Без журналирования и обработки сбоев** — нет ретраев при ошибках сети/диска, нет логов в файл, только вывод в консоль.

## 🛠 Как запустить демо

```bash
pip install watchdog cryptography
python main.py
```

После запуска:
1. Скрипт создаст папку `./MyDocuments` (рабочая зона) и `./NetworkCloudStorage` (эмулятор хранилища), если их ещё нет.
2. Измените (не просто создайте, а сохраните изменения в) любой файл внутри `MyDocuments`.
3. В `NetworkCloudStorage` появится файл `<имя>.enc` — зашифрованная копия, которую нельзя прочитать в текстовом редакторе.

> Обратите внимание: при первом сохранении нового файла синхронизация может не сработать — обработчик слушает `on_modified`. Измените файл повторно (например, добавьте и сохраните строку), чтобы увидеть срабатывание.

## 💼 Коммерческая доработка

Репозиторий содержит только базовое ядро логики (агент шифрования "на лету"). Под задачи конкретного бизнеса я дорабатываю это до полноценного решения. Возможные направления доработки:

- **Управление ключами** — безопасное хранение, ротация и резервирование ключа шифрования (без этого расшифровка данных невозможна в принципе).
- **Модуль расшифровки** — отдельная утилита/панель для администратора, позволяющая восстанавливать файлы из `.enc`-копий.
- **Реальные протоколы передачи** — интеграция с SFTP/FTP или объектными хранилищами (Amazon S3, Backblaze B2, Яндекс Object Storage и др.) вместо локальной папки-эмулятора.
- **Расширенный мониторинг** — рекурсивный обход подпапок, обработка создания/удаления/переименования файлов.
- **Планировщик и ротация** — расписание полных/инкрементальных бэкапов, автоматическая очистка старых версий.
- **Уведомления** — интеграция с Telegram/Slack/email для отчётов о статусе синхронизации и ошибках.
- **Логирование и отказоустойчивость** — ретраи при сетевых сбоях, структурированные логи, мониторинг состояния агента.

Если нужна система синхронизации и шифрования файлов под ваши задачи — пишите в личные сообщения на бирже, обсудим архитектуру и объём работ под конкретную инфраструктуру.
